# 小智（Xiaozhi）全量内容迁移设计

## 1. 背景

当前小智的新站生产底座已经完成：

- 基于 `Next.js + TypeScript + React + App Router`
- 已具备首页、学科页、模块页、知识点页与 demo shell
- 已打通一条完整的示范性迁移链路：`数学 / 函数 / 二次函数`
- 已具备基础 SEO、CI、桌面端与手机端兼容验证能力

与此同时，旧原型内容仍主要保存在 `legacy/prototype-v1/` 中，尚未系统迁入现行生产站点。

截至 2026-07-03，旧原型包含：

- `4` 个学科入口：数学、物理、化学、英语
- `17` 个模块页
- `89` 个静态模块/单元页面文件
- `62` 个旧演示器目录
- 一批历史数学推导、图形引擎与教学卡片逻辑

本设计聚焦下一阶段核心任务：把旧原型中的全部内容迁移到现行生产站点，并在迁移过程中完成必要重构，而不是简单搬运。

## 2. 目标

### 2.1 总目标

把 `legacy/prototype-v1/` 中的全部学科结构、模块信息、知识点内容和演示器能力，迁移为新站中的正式生产资产。

### 2.2 交付结果

迁移完成后，新站需要满足以下结果：

- 四个学科的主线结构都在新站中可访问
- 所有旧知识点都在新站里有正式的内容入口
- 高价值知识点拥有正式可交互 demo
- 其余知识点至少拥有规范化内容页与明确状态
- 所有内容遵循统一 slug、内容模型、元数据和 SEO 规则
- 桌面端浏览器与手机端浏览器都能稳定访问
- 旧原型从“运行主体”降级为“迁移归档”

## 3. 迁移策略

本次采用以下策略组合：

- `迁移同时重构`
- `允许适度重组`
- `分层迁移`

这意味着：

- 不直接把旧原型页面原样接入新站
- 保留原有学科主线，不做大幅改科目方向
- 允许对命名混乱、粒度过碎、表达重复的单元做小幅合并或拆分
- 先统一内容模型与站点骨架，再批量迁入内容和 demo

## 4. 迁移原则

### 4.1 学科主线稳定

以下主线保持不变：

- 数学：函数、三角、数列、解析几何、立体几何、集合与逻辑
- 物理：运动、受力、电学、功与能量、振动与波动
- 化学：化学语言、反应原理、元素化合物、实验现象
- 英语：句法结构、词根词缀、阅读路径、写作表达

### 4.2 内容模型先行

迁移不是“把 html 文件放进新路由”，而是先将旧内容抽象成统一数据模型，再生成页面。

### 4.3 适度重组

允许以下类型的小幅重构：

- 同一主题下命名重复的单元归并
- 表达过碎但必须连读的知识点合并为一个单元
- 过大的知识点拆成“内容页 + demo 页”或“总览页 + 单元页”

不允许以下类型的重构：

- 改变学科主线
- 把旧学科主题拆成完全不同的课程结构
- 在没有教学收益的前提下重新命名全部单元

### 4.4 兼容性优先

每一批迁移必须同时考虑：

- 电脑端浏览器访问
- 手机端浏览器访问
- 首屏结构清晰
- demo 交互区域在窄屏上可用

### 4.5 迁移可追踪

所有内容迁移必须有：

- 映射台账
- 迁移状态
- 验收标准
- 对应测试
- 分批提交记录

## 5. 现状盘点

### 5.1 新站现状

当前新站正式内容范围较小：

- 学科入口：4 个
- 模块骨架：4 个
- 正式 demo：1 个（`quadratic-function`）
- 正式内容文件极少，尚未形成完整知识库

### 5.2 旧原型内容范围

旧原型中已经形成了较完整的内容骨架：

- 数学：函数、三角函数、数列、解析几何、立体几何、集合
- 物理：运动、受力、电学、功能量、波动
- 化学：化学语言、反应、元素化合物、实验
- 英语：句法、词汇网络、阅读路径、听力、写作、语法填空

旧原型还包含三类高价值资产：

1. 内容结构资产
2. 教学表达资产
3. 演示逻辑资产

其中演示逻辑资产主要分布在：

- `legacy/prototype-v1/assets/math-viz-core.mjs`
- `legacy/prototype-v1/demos/shared/`
- `legacy/prototype-v1/demos/*`

迁移后的正式站点统一采用以下层级：

- 首页
- 学科页
- 模块页
- 知识点页
- 可选 demo 区块

### 6.1 页面职责

#### 首页

- 学习方式说明

#### 学科页

- 学科主线介绍

## 7. 内容模型设计

正式迁移后至少需要以下实体：

- `Subject`
- `Module`
- `KnowledgeUnit`
- `DemoDefinition`
- `LearningPath`
- `MigrationManifestRecord`

### 7.1 Subject

负责学科元信息：

- `id`
- `slug`
- `nameZh`
- `nameEn`
- `summary`
- `theme`
- `seo`

### 7.2 Module

负责主线模块元信息：

- `id`
- `subjectId`
- `slug`
- `title`
- `summary`
- `highlights`
- `order`
- `status`
- `seo`

### 7.3 KnowledgeUnit

负责知识点页的标准化内容：

- `id`
- `subjectId`
- `moduleId`
- `slug`
- `title`
- `titleEn`
- `summary`
- `difficulty`
- `learningGoals`
- `coreTakeaways`
- `keywords`
- `relatedUnits`
- `status`
- `demoIds`
- `migrationSource`
- `seo`

### 7.4 DemoDefinition

负责统一接入正式演示器：

- `id`
- `unitId`
- `kind`
- `renderer`
- `defaultParams`
- `controls`
- `presets`
- `explanation`
- `mobileBehavior`

### 7.5 MigrationManifestRecord

负责迁移台账：

- `legacyType`
- `legacyPath`
- `legacyTitle`
- `targetSubject`
- `targetModule`
- `targetUnitSlug`
- `action`
- `status`
- `priority`

其中 `action` 仅允许：

- `migrate-as-is`
- `merge`
- `split`
- `rewrite-demo`
- `content-only`
- `archive-only`

## 8. 迁移分层

### 8.1 第一层：结构层

目标：

- 先让所有学科、模块、知识点在新站中都有正式入口
- 即使 demo 尚未完成，也要先形成完整知识地图

交付：

- 全量学科、模块、知识点注册
- 正式路由可访问
- 占位状态可读

### 8.2 第二层：内容层

目标：

- 把旧原型的摘要、学习目标、知识结构与教学文案转为正式内容源

交付：

- MDX 或结构化内容文件
- 统一的知识点页模板
- 相关链接、前置知识、延伸路径

### 8.3 第三层：交互层

目标：

- 把旧 demo 重写进正式演示框架

交付：

- 正式 `DemoDefinition`
- TypeScript 化数学/物理/化学/英语逻辑
- 桌面端与手机端都可用的交互布局

## 9. 学科迁移顺序

推荐顺序如下：

1. 数学
2. 物理
3. 化学
4. 英语

### 9.1 数学优先

原因：

- 与当前 demo 引擎最接近
- 已有二次函数示范迁移
- `math-viz-core.mjs` 中有大量可重写为 TypeScript 的逻辑

### 9.2 物理第二

原因：

- 旧原型内容成熟
- 图形化和过程性很强
- 与现有可视化模型契合度高

### 9.3 化学第三

原因：

- 内容结构相对清晰
- 需要更多流程图、粒子图和状态图表达
- 交互形态会比数学/物理更异构

### 9.4 英语第四

原因：

- 结构表达丰富，但未必都需要“连续参数交互”
- 更适合与内容型页面和轻交互模块结合

## 10. 适度重组规则

迁移时采用以下规则：

### 10.1 可合并

- 同一模块内仅是同类表达变体的微小单元
- 必须并读才能形成完整认知的知识点
- 旧命名混乱但实质相同的单元

### 10.2 可拆分

- 一个旧单元同时承担“总览 + 细节 + 交互”三种职责
- 页面内容过大，不利于手机端阅读
- 一个单元内含多个独立认知目标

### 10.3 不重组

- 已经是清晰、独立且高价值的核心知识点
- 已形成稳定 demo 逻辑的单元
- 只因命名习惯不同、不影响认知边界的内容

## 11. 工程方案

### 11.1 内容源

- 学科、模块、路径：`ts`
- 知识点正文：`mdx`
- demo 配置：`ts`
- 迁移台账：`ts` 或 `json`

### 11.2 正式目录目标

建议新增或扩展：

- `content/subjects.ts`
- `content/modules.ts`
- `content/units/<subject>/<module>/<unit>.mdx`
- `content/demos/*.ts`
- `content/learning-paths/*.ts`
- `content/migration-manifest.ts`

### 11.3 引擎重写方向

旧原型中的共享逻辑不直接复制：

- 数学函数采样与几何计算逻辑重写到 `engine/core/`
- 共享教学卡片结构抽象为组件或 schema
- 特定 demo 的旧插件写法转换为统一 `DemoDefinition`

## 12. 验收标准

迁移计划完成后，每个阶段都必须满足对应验收：

### 12.1 结构层验收

- 全量学科可访问
- 全量模块可访问
- 全量知识点有正式路由
- 不再依赖旧 html 文件作为站点入口

### 12.2 内容层验收

- 每个知识点至少有摘要、学习目标、关键规律
- 页面内容在手机端可读
- 面包屑和相关链接完整

### 12.3 交互层验收

- 高优先级知识点具备正式 demo
- 参数变化、讲解面板和图形区域在桌面端与手机端都可用
- 关键 demo 具备单元测试和端到端验证

### 12.4 收口验收

- 旧原型只保留归档职责
- README 与迁移说明同步更新
- 新站成为唯一正式运行入口

## 13. 风险与应对

### 13.1 内容量过大

风险：

- 一次性迁移全部内容会导致计划失控

应对：

- 采用分层迁移
- 每批迁移可独立验收与提交

### 13.2 旧逻辑风格不统一

风险：

- 直接搬运旧 demo 会污染新架构

应对：

- 只迁移“知识逻辑”，不迁移旧脚本组织方式
- 强制重写为正式 TypeScript 接口

### 13.3 手机端可用性不足

风险：

- 旧 demo 以桌面端为主，窄屏下容易不可用

应对：

- 所有新 demo 默认 mobile-first 布局
- 在验收中强制包含手机端检查

### 13.4 命名与粒度失控

风险：

- 迁移中不断改名、改单元边界，会导致路线混乱

应对：

- 先做迁移台账
- 先定 slug 与 action，再写内容和代码

## 14. 结论

本次全量内容迁移不是一次简单搬迁，而是一轮“生产化内容重建”。

推荐实施方式是：

- 先建台账
- 再补骨架
- 再迁内容
- 最后重写 demo

这样可以保证小智最终得到的是一个可长期维护、兼容电脑端与手机端、内容完整且演示能力可持续扩展的正式产品站点，而不是新壳套旧站。
