# 小智（Xiaozhi）生产级重构设计文档

## 1. 背景与目标

小智（Xiaozhi）是一个面向高中学习的知识可视化项目，目标是把抽象、零散、容易停留在结论层的高中知识点，转化为图形化、可交互、可讲透的学习体验，帮助学生理解知识背后的结构、变化规律与底层原理。

当前目录中已经存在一版前端演示工程。现有工程已经具备以下基础：

- 已形成数学、物理、化学、英语四门学科的内容骨架
- 已沉淀一批知识点页面与交互演示雏形
- 已有统一演示引擎的初步结构
- 已存在一定数量的自动化测试

但它整体上仍偏向原生 `HTML/CSS/JS` 的演示型项目，尚未达到生产级项目的工程化标准。接下来需要把它升级为一个可持续开发、可稳定上线、可在 GitHub 上规范协作与交付的正式产品代码库。

本次设计聚焦第一阶段：完成站点前台的生产级重构底座。

## 2. 第一阶段范围

### 2.1 第一阶段要实现的内容

第一阶段将小智定义为一个内容型高中知识可视化学习网站，重点建设以下能力：

- 正式的网站前台架构
- 首页、学科页、模块页、知识点页、交互演示页
- 统一的内容模型与路由体系
- 全新的 React 化交互演示引擎
- 电脑端与手机端浏览器兼容的响应式体验
- 基础 SEO 能力
- 自动化测试体系
- CI/CD 与 GitHub 协作底座
- 可部署、可持续发布的工程结构

### 2.2 第一阶段明确不做的内容

为控制范围，以下能力不纳入第一阶段交付：

- 重后台 CMS
- 完整用户体系
- 付费体系
- 教师端、运营端
- 复杂学习档案与长期行为分析

这些能力需要在未来阶段基于当前架构继续扩展，但不应挤占第一阶段的重构资源。

## 3. 产品定位

第一阶段的小智对外是一个内容型学习网站，而不是完整平台产品。

它的核心价值不是“题库”或“刷题工具”，而是通过图形化与交互化方式，帮助高中生对关键知识点形成结构理解。页面需要围绕“讲透一个知识点”来组织，而不是仅做知识罗列。

这一定位带来几个直接要求：

- 页面内容要可阅读、可讲解、可复习
- 交互演示不是装饰，而是核心教学方式
- 网站要同时适合电脑端与手机端访问
- 路由、SEO、内容组织都要面向公开访问场景设计

## 4. 技术决策

### 4.1 技术路线

第一阶段采用：

- `Next.js`
- `TypeScript`
- `React`
- `App Router`

并在该技术栈下同时重构：

- 网站页面层
- 内容组织层
- 交互演示引擎层

本次重构不走“旧原生页面长期保留”的双轨方案，而是以统一技术栈建立新的正式工程。

### 4.2 选择这一方案的原因

选择 `Next.js + TypeScript + React + App Router` 的主要原因如下：

- 适合内容型站点的静态生成、SEO 与页面性能优化
- 支持后续继续扩展为更完整的平台
- 便于建立统一页面层、统一内容模型和统一交互引擎
- 更适合 GitHub 协作、工程规范和自动化测试建设
- 可以在统一项目里同时管理内容页与复杂交互组件

## 5. 整体架构

第一阶段建议在一个统一的 Next.js 工程中，清晰拆分为四层：

1. 页面层：负责路由、布局、SEO、静态生成和数据装配
2. 业务功能层：负责学科、知识点、演示播放器等业务模块
3. 内容层：负责学科、模块、知识点与演示配置等内容数据
4. 引擎层：负责图形化交互演示的状态、渲染与控制逻辑

推荐目录方向如下：

```text
app/
components/
features/
  subjects/
  knowledge/
  demo-player/
content/
  subjects/
  modules/
  units/
  demos/
engine/
lib/
tests/
```

### 5.1 页面层职责

页面层只负责：

- 路由定义
- 页面装配
- 元信息生成
- 服务端或静态数据读取
- 页面级布局结构

页面层不直接承担复杂演示逻辑。

### 5.2 内容层职责

内容层负责维护知识结构与页面内容，不直接关心渲染框架细节。其目标是让“知识内容”与“页面表现”解耦，为未来接 CMS 留出空间。

### 5.3 引擎层职责

引擎层是小智的核心差异化资产，负责把知识点演示标准化，而不是继续以“每个页面一套独立脚本”的方式扩张。

## 6. 信息架构

网站整体信息架构建议采用四层结构：

- 首页
- 学科页
- 模块页
- 知识点页 / 演示页

### 6.1 首页

首页承担以下职责：

- 品牌与项目定位表达
- 学科入口
- 重点学习主线推荐
- 项目价值与学习方式说明

### 6.2 学科页

学科页用于组织某一门学科的主线结构，例如数学、物理、化学、英语。每个学科页需要展示：

- 学科简介
- 主线模块
- 重点知识入口
- 推荐学习路径

### 6.3 模块页

模块页承接学科内部的主题主线，例如函数、解析几何、受力分析、句子结构。模块页负责：

- 聚合某条主线下的知识点
- 展示该主线内部的结构关系
- 引导用户进入具体知识点

### 6.4 知识点页

知识点页是最核心的页面单元，应围绕“讲透一个知识点”来组织，至少包含：

- 知识点简介
- 学习目标
- 关键规律
- 图示与讲解
- 交互演示器
- 易错点与延伸阅读

## 7. 路由设计

路由应采用语义化结构，避免延续以静态 `html` 文件为中心的命名方式。

推荐示例：

```text
/
/subjects/math
/subjects/physics
/subjects/math/functions
/subjects/math/functions/quadratic-function
```

这一结构需要支持：

- SEO 友好 URL
- 面包屑导航
- 学科内跳转
- 后续站内搜索与推荐系统

## 8. 内容模型设计

第一阶段内容模型至少包含四类核心实体：

1. `Subject`
2. `Module`
3. `KnowledgeUnit`
4. `Demo`

### 8.1 Subject

代表学科，例如数学、物理、化学、英语。

建议字段：

- `id`
- `slug`
- `nameZh`
- `nameEn`
- `summary`
- `theme`
- `seo`

### 8.2 Module

代表学科内部主线模块，例如函数、解析几何、受力分析、句子结构。

建议字段：

- `id`
- `subjectId`
- `slug`
- `title`
- `summary`
- `highlights`
- `order`
- `seo`

### 8.3 KnowledgeUnit

这是最核心的内容实体，未来应成为知识组织与页面渲染的主要原子。

建议字段：

- `id`
- `slug`
- `title`
- `titleEn`
- `subjectId`
- `moduleId`
- `summary`
- `learningGoals`
- `coreTakeaways`
- `keywords`
- `prerequisites`
- `relatedUnits`
- `demoIds`
- `seo`

正文内容建议使用 `MDX` 表达，以同时支持结构化元信息与图文内容扩展。

### 8.4 Demo

代表挂载在知识点页中的交互演示定义。

建议字段：

- `id`
- `unitId`
- `title`
- `type`
- `defaultParams`
- `presets`
- `viewport`
- `mobileLayout`
- `teachingHints`

## 9. 内容存储策略

第一阶段不引入 CMS，先采用代码内内容源。

推荐方式：

- 结构化元数据使用 `ts` 或 frontmatter
- 长文本内容使用 `mdx`
- 演示配置使用 `ts`

推荐目录方向：

```text
content/
  subjects/*.ts
  modules/*.ts
  units/**/*.mdx
  demos/*.ts
```

这种方式的优势：

- 开发与重构效率高
- 与 GitHub 版本控制天然协同
- 容易做类型约束和测试校验
- 未来迁移 CMS 时边界清晰

## 10. 交互演示引擎重构方案

### 10.1 重构目标

新的演示引擎需要从“页面脚本集合”升级为“可复用的统一演示系统”。

每个知识点不再自己维护一整页散装交互脚本，而是通过统一 schema 描述：

- 参数
- 默认值
- 预设场景
- 图层
- 讲解信息
- 移动端布局规则

### 10.2 引擎分层

建议拆分为以下部分：

1. `demo definition`
2. `state model`
3. `renderer`
4. `control panel`
5. `explanation panel`
6. `responsive shell`

### 10.3 demo definition

每个演示器通过配置定义自身：

- 可调参数
- 初始状态
- 教学预设
- 可视化图层
- 对应讲解文案

### 10.4 state model

统一管理：

- 参数状态
- 预设切换
- 当前视窗
- 动画状态
- 移动端折叠状态

### 10.5 renderer

负责可视化输出。图形层优先采用 `SVG + Canvas` 混合策略：

- 坐标系、标注、规则图形优先用 `SVG`
- 高频重绘、动画部分优先用 `Canvas`

### 10.6 control panel

负责：

- 参数滑杆
- 开关
- 预设按钮
- 视图切换

### 10.7 explanation panel

负责把“图像变化意味着什么”同步表达为教学信息，而不是只显示数值变化。

### 10.8 responsive shell

负责演示器在桌面与移动端之间切换布局：

- 桌面端以多栏布局为主
- 移动端以单栏折叠布局为主

## 11. 响应式与多端兼容策略

小智第一阶段必须同时兼容电脑端浏览器和手机端浏览器。

### 11.1 设计原则

- 采用 `mobile-first`
- 小屏先保证可用，再向大屏增强
- 页面布局规则统一
- 演示页具备独立的响应式外壳

### 11.2 页面适配要求

首页、学科页、模块页、知识点页、演示页都需要在常见手机宽度和桌面宽度下正常访问、阅读与操作。

### 11.3 演示器适配要求

移动端不能简单缩小桌面布局，需要做专门适配：

- 参数区可折叠
- 画布区域优先展示
- 讲解内容可上下堆叠
- 支持触控操作

桌面端推荐布局：

- 左侧或顶部为知识概览
- 中间为主要演示画布
- 侧边为参数控制与讲解

移动端推荐布局：

- 顶部为标题与关键结论
- 中部为主画布
- 下部为折叠控制面板与讲解卡片

## 12. 性能策略

### 12.1 页面性能目标

第一阶段应把页面加载性能视为正式指标，特别是手机网络环境下的访问体验。

### 12.2 内容与交互分离

需要将静态内容与重交互模块分离处理：

- 学科页、模块页、知识点正文尽量静态生成
- 演示器采用按需加载
- 重交互代码不阻塞首屏阅读内容

### 12.3 资源策略

统一管理：

- 图片与插图资源
- 字体与样式加载
- 交互脚本切分

目标是保证：

- 首屏核心内容优先呈现
- 手机端访问不因演示器阻塞主要阅读
- 演示页滚动与交互保持流畅

## 13. SEO 策略

第一阶段既然是内容型网站，就必须建设基础 SEO 能力。

每个页面应具备：

- 独立 `title`
- 独立 `description`
- 规范 URL
- Open Graph 信息
- 面包屑信息

知识点页还应具备：

- 清晰的知识点摘要
- 适合搜索引擎理解的结构化文本
- 相关推荐与上下游知识链接

## 14. 测试策略

测试体系建议分为四层：

### 14.1 内容模型测试

用于校验：

- slug 唯一性
- 内容字段完整性
- 实体引用有效性
- 演示配置引用有效性

### 14.2 引擎单元测试

覆盖：

- 参数计算
- 状态变更
- 预设切换
- 渲染配置

### 14.3 页面集成测试

覆盖：

- 关键路由渲染
- 页面导航
- 内容渲染
- 响应式布局切换

### 14.4 端到端测试

至少覆盖以下主链路：

- 首页进入学科页
- 学科页进入知识点页
- 知识点页打开演示器
- 演示器参数切换
- 手机视口访问关键页面

## 15. GitHub 协作与交付

第一阶段需要把 GitHub 协作方式一并设计进去。

建议：

- `main` 作为稳定分支
- 通过 Pull Request 合并变更
- PR 检查至少包含 `lint`、`typecheck`、`test`、`build`
- 自动生成预览环境
- 主分支具备自动发布能力

这样可以保证仓库从一开始就具备正式项目的协作体验，而不是仅作为代码存放位置。

## 16. 迁移原则

尽管本次选择的是“页面层与交互层一起重构”的激进路线，迁移过程仍应遵循阶段化原则，而不是一次性无序重写。

建议迁移时遵循以下原则：

- 先建立新工程骨架
- 先完成核心信息架构与内容模型
- 优先迁移首页、一个学科主线、一个代表性知识点链路
- 优先沉淀通用演示引擎能力
- 再逐步迁移其他学科与知识点

这样可以控制风险，并持续验证架构是否真正适合小智的内容形态。

## 17. 非目标与风险

### 17.1 非目标

本次不追求：

- 一次性完成所有旧内容迁移
- 一次性完成所有学科的完整 UI 统一
- 一次性补全未来平台化能力

### 17.2 主要风险

需要重点关注的风险包括：

- 重构范围过大导致交付周期失控
- 演示引擎重写初期效率低于预期
- 旧内容迁移时字段结构不统一
- 移动端交互适配复杂度高于普通内容站

因此后续 implementation plan 必须分阶段拆解，并优先建立最小可验证路径。

## 18. 结论

小智第一阶段应重构为一个基于 `Next.js + TypeScript + React + App Router` 的内容型高中知识可视化学习网站。

它需要：

- 同时兼容电脑端和手机端浏览器
- 以“首页 -> 学科页 -> 模块页 -> 知识点页 -> 演示页”的结构组织内容
- 以 `KnowledgeUnit` 为核心内容原子
- 以统一 React 化演示引擎承接核心教学交互
- 在第一阶段一次性补齐 SEO、性能、测试、CI/CD 和 GitHub 协作底座

这一方案既能支撑正式上线，也能为未来继续扩展用户系统、CMS、搜索、推荐和平台能力打下稳定基础。
